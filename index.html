REM Resident Externalized Memory — a local agent's working memory, consolidated on a background co-processor instead of stealing cycles from the model.

While the big model is awake on the iGPU, the NPU sleeps on it — compacting the conversation into a compact, fact-preserving context on separate silicon, off the GPU's critical path and inside a measured contention budget.

The placement question

Does running on the NPU actually leave the GPU alone?

The whole premise is that memory maintenance can run next to the foreground model without slowing it down. So we measured it: the same compaction workload on the NPU vs. a control on spare CPU cores, each running concurrently with an interactive iGPU model. Pooled over 3 runs × N=20 = 60 decode samples, measured with the xdna-top monitor.

The honest read: this is a placement win — throughput, perf/watt, and keeping the foreground free — not a "contention-free" claim. The NPU still shares the unified memory bus; it just contends no worse than spare CPU cores while doing ~3× the work for ~3× less energy.

The keep-up question

Can it drain memory faster than the agent fills it?

A background memory layer is only viable if it keeps up: the NPU must consolidate context at least as fast as the conversation grows, or the backlog overflows and the turn fails. The compactor drains at ~73 tok/s. Here's the headroom against three agent speeds (from throughput_probe.json).

Why async matters here: compaction runs in a background thread, so a ~19 s pass is never a per-turn stall — the foreground turn returns immediately. Async only helps while the drain keeps up; if it ever falls behind a fast agent, the context backlog can hit a hard cap and fail the turn. That's why the keep-up margin, not just latency, is the number that matters.

The sustained-load question

Does it cook the chip under continuous use?

There's no discrete NPU temperature sensor — XDNA2 lives on the same monolithic SoC die — so we track the SoC control temp (k10temp Tctl) as the proxy. Over a 192-minute continuous compaction run: mean 41 °C, max 61 °C, dead flat. Tens of degrees below any throttle point.

How it works

A tiered memory sidecar in front of the model

A sidecar sits in front of the foreground model's chat endpoint. It keeps memory in tiers and assembles a budget-bounded context for each request. When the verbatim tier crosses a threshold, compaction fires on the NPU in the background — extracting load-bearing facts from the oldest span and replacing it with a summary, never blocking the foreground turn.

Honest status

What's demonstrated vs. what's open

This is a feasibility study. The placement pillars hold on real hardware; the quality work is a clear development path, not a finished result — and that's where we'd welcome help.

✓ Demonstrated on a Strix Halo box

  • Off the critical path. Compaction runs in a background thread; the turn returns without waiting (~18–24 s pass is never a per-turn stall).
  • It keeps up. ~73 tok/s drain — above realistic agent context-growth, so the backlog stays bounded.
  • It runs cool. 192 min sustained: mean 41 °C / max 61 °C, no drift.
  • The placement is justified. ~3.8% iGPU decode loss vs ~4.8% on CPU, at ~3× throughput and ~3× perf/watt.
  • The compaction mechanism runs. It extracts facts and rewrites the oldest span at ~10× compression (9.9× measured). Whether buried facts reliably survive is an open quality question, not a settled result.

▸ To be developed

  • Small-model JSON robustness. The on-NPU model sometimes emits malformed JSON during fact extraction; those facts get dropped. A repair/retry pipeline exists but isn't bulletproof.
  • Fact identity / supersession. When a value is updated ("I moved to Denver"), REM can still surface the stale one. Path forward: embedding-based identity (match by meaning, not string).
  • Quality vs. naive baseline is unproven & budget-conditional. At a generous budget, simple truncation can match or beat REM. The one judged battery run so far was invalidated (budget too generous to test the hypothesis), and on that subset REM's accuracy was lower than truncation's (0.4 vs 0.6). A valid tight-budget win — plus JSON robustness — is the open path to a defensible quality number.
