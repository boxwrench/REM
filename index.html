The core idea

Let the task grow. Keep injected context flat.

A naive loop appends every observation and answer to the next prompt, so context compounds until the model's window blows. REM files bulky history outside the model and returns a bounded slice of relevant state each step.

Brain

The model chooses

It reads what is on its desk and decides the next action. It does not need every raw page it has ever seen.

Notebook

REM remembers

It compacts bulky history, keeps load-bearing facts, and selects a fit-to-budget view for the next step.

World

The environment responds

It returns files or tool results, accepts actions, tracks state, and knows how to score the task.

Canonical example: a long water-quality audit must retain an early limit while opening report after report. Read the full Lesson 4 worked example.

The placement question

Does running on the NPU actually leave the GPU alone?

The same compaction workload ran on the NPU and on spare CPU cores, each concurrently with an interactive iGPU model. Across 3 runs × N=20, NPU placement imposed a slightly smaller foreground cost.

What the result means

a placement result, not a latency trick

Background memory work can use otherwise-idle NPU compute while leaving the iGPU focused on the foreground model.

The shared-memory caveat

separate compute does not mean separate bandwidth

The NPU and iGPU still share one unified-memory bus, so the measured decode loss is real and belongs in the operating budget.

The honest read: this is a placement win, not a “contention-free” claim. The NPU shares the unified-memory bus; it simply imposed a smaller measured decode loss than the CPU control.

The keep-up question

Can it drain memory faster than the agent fills it?

A background memory layer is viable only if it keeps up. The committed compactor probe drains at ~73 tok/s while the example transcript grows at ~30 tok/s: a ~2.5× keep-up margin.

How it works

A tiered memory sidecar in front of the model

A sidecar sits in front of the foreground model's chat endpoint. It keeps memory in tiers and assembles a budget-bounded context for each request. When the verbatim tier crosses a threshold, compaction fires on the NPU in the background — extracting load-bearing facts from the oldest span and replacing it with a summary, never blocking the foreground turn.

Verbatimrecent

The most recent turns, kept word-for-word. Highest fidelity, highest token cost.

→

Summaries

Older spans compacted into smaller prose summaries by the NPU, while load-bearing facts are filed separately.

Load-bearing facts extracted and filed so they survive compaction even after the prose is gone.

Honest status

Read-side win shipped. Horizon proof next.

REM has reoriented around context budget over agent horizon. The read-side is working; the agent-horizon harness is the next milestone, and no horizon multiplier has been measured yet.

✓ Demonstrated on a Strix Halo box

  • Bounded read assembly is shipped. The sidecar selects a fit-to-budget view instead of injecting the whole store.
  • Compaction keeps up. ~73 tok/s drain against ~30 tok/s example growth gives a ~2.5× margin.
  • Prompt work falls. RSB-3 shrank 47.6%, from 27,386 to 14,341 tokens; the full prefill accounting still saved +3.77 s.
  • Placement has a measured budget. ~3.8% iGPU decode loss versus ~4.8% for the CPU control across 3 runs × N=20.

▸ Next milestone

  • Measure the horizon multiplier. Replay identical long tasks through naive, truncate, and REM arms; this result does not exist yet.
  • Prove graceful stop. Freeze a state card and hand off before a hard context failure.
  • Confirm OpenEnv fit. Validate fat-observation, ≥100-step task worlds before adopting the interface.
  • Hold correctness as a floor. More steps count only if the task grader does not regress.
